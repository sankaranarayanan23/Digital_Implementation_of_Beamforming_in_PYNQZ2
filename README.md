## 1. Project Overview

This project demonstrates a **complete hardware implementation pipeline** for digital beamforming using a 4-element antenna array. The beamforming algorithm computes element-wise phase weights using the CORDIC algorithm and sums the weighted signals to steer a beam toward a desired angle.

### What Was Built

A fixed-point, HDL-compatible beamforming engine was designed in **MATLAB/Simulink**, exported as a **custom Verilog IP core** via HDL Coder, integrated into a **Zynq PS+PL system** using Xilinx Vivado, and finally deployed on a **PYNQ-Z2 FPGA board**. Control and verification were done over Python using the PYNQ framework.

### Key Achievements

- Full fixed-point (`sfix16_en12`) beamformer implemented and verified in simulation
- Auto-generated synthesizable Verilog from the Simulink model using MATLAB HDL Coder
- Custom IP integrated into a Zynq block design with AXI4-Lite memory-mapped control
- Successfully deployed on PYNQ-Z2 and verified with four phase-shifted sine wave inputs
- Achieved perfect constructive interference with a combined output amplitude of 2.0 (the theoretical maximum for 4 elements with unit amplitude 0.5 each)

## 2. System Architecture

The full system flow is:

```
Python (PYNQ)  ─── AXI4-Lite ───►  Beamformer IP (PL)
     │                                      │
     │  writes phase-shifted complex         │  reads output
     │  inputs via MMIO registers            │  combined beam
     ▼                                      ▼
ZYNQ PS (ARM Cortex-A9)        Custom Subsystem IP (Verilog)
     │                                      │
     └────── AXI Interconnect ──────────────┘
                    │
           Processor System Reset
```

### Block Diagram (Vivado)

The Vivado block design contains four main components:

| Block | Role |
|---|---|
| **ZYNQ7 Processing System** | Master controller; provides 50 MHz FCLK; runs Python application |
| **AXI Interconnect** | Memory-mapped bridge between ARM and PL; routes AXI4-Lite traffic |
| **Subsystem IP (Beamformer)** | Slave peripheral; contains the HDL-generated beamforming logic |
| **Processor System Reset** | Takes raw reset from the PS and synchronizes all PL components cleanly |

## 5. Theory: Phased Array Beamforming

### Basic Concept

In a phased array, signals from multiple antennas are combined with specific **phase shifts** to constructively add signals arriving from a desired direction and cancel signals from other directions.

For a 4-element Uniform Linear Array (ULA) with element spacing `d` and target steering angle `θ`, the phase shift applied to element `n` is:

```
φ_n = (2π / λ) × n × d × sin(θ)
```

Where:
- `λ` is the signal wavelength
- `d` is the inter-element spacing (typically `λ/2`)
- `n` is the element index (0, 1, 2, 3)

### Fixed-Point Representation

All arithmetic in this design uses the **sfix16_en12** format:
- **16 bits** total word length
- **12 bits** for the fractional part (i.e., 4 integer bits + 12 fractional bits)
- Range: approximately `-8.0` to `+7.9997`
- Precision: `2^-12 ≈ 0.000244`

This format is chosen for efficient FPGA implementation — it fits in DSP48 slices on the Zynq FPGA and avoids floating-point hardware overhead.

### CORDIC for Phase Rotation

Phase weights are computed using the **CORDIC (COordinate Rotation DIgital Computer)** algorithm. CORDIC computes `sin` and `cos` using only bit shifts and additions — no multipliers needed — making it ideal for FPGA deployment.

For each antenna element, the complex input signal `(real, imag)` is rotated by angle `φ_n`:
```
real_out = real_in × cos(φ_n) − imag_in × sin(φ_n)
imag_out = real_in × sin(φ_n) + imag_in × cos(φ_n)
```

The four rotated outputs are then summed to form the beamformed output.

### Inside the Beamformer Subsystem

The subsystem contains the HDL-compatible logic:

| Block | HDL Library | Purpose |
|---|---|---|
| `Gain` | Simulink HDL | Apply fixed-point gain |

### Why Only Built-in Simulink Blocks?

HDL Coder can only generate Verilog/VHDL for blocks that have an **HDL implementation**. All blocks inside the subsystem must be from the HDL-compatible library. Blocks like MATLAB Function, interpreted scopes, or floating-point blocks will cause HDL generation to fail.

## 7. Step 2 — HDL Subsystem Creation

Before generating HDL code, the beamforming logic must be isolated in a Simulink Subsystem. The Subsystem becomes the **Design Under Test (DUT)** — only blocks inside it are converted to Verilog.

### Creating the Subsystem (if building from scratch)

1. In Simulink, **select all blocks** that form the beamforming logic (Gain, CORDIC, Product, Add blocks).
2. Right-click on the selection → **Create Subsystem from Selection**.
3. Name the subsystem `Beamformer` (or `Subsystem`).
4. Verify inputs and outputs are correctly connected to the subsystem ports.

### Why Use a Subsystem?

- HDL Coder generates code **only for subsystems**, not for the whole model.
- Only blocks inside the subsystem are converted into Verilog.
- The testbench (sine wave sources, scopes, phase delay blocks) stays outside — it is used for simulation only and is not synthesized.
- This isolates the hardware logic from testbench signals cleanly.

### Verify HDL Compatibility

Before running HDL Coder, check that all blocks inside the subsystem are HDL-compatible:

1. Go to **Apps** → **HDL Coder**.
2. In the HDL Coder tab, click **Run HDL Checks**.
3. Resolve any errors or warnings (typically floating-point blocks, unsupported operations, or data type mismatches).

## 9. Step 4 — Vivado Block Design with Beamformer IP

### Add the IP to the Vivado Repository

1. Open **Xilinx Vivado** and create a new RTL project targeting `xc7z020clg400-1`.
2. Go to **IP Catalog** → **Manage IP** → **Add Repository**.
3. Browse to the `ip_core/` folder generated by HDL Coder.
4. Click **Select** — Vivado will detect and register the beamformer IP.

### Create the Block Design

1. Under **Flow Navigator** → **IP Integrator**, click **Create Block Design**.
2. Name it (e.g., `beamform_system`).

### Add and Configure Blocks

#### Add ZYNQ7 Processing System

1. Click the **+** button → search for `ZYNQ7 Processing System` → double-click to add.
2. Click **Run Block Automation** (accept defaults — this configures DDR and fixed IO).
3. Double-click the Zynq block → in **Clock Configuration**, ensure `FCLK_CLK0` is set to **50 MHz**.
4. In **PS-PL Configuration** → **AXI Non Secure Enablement** → enable **M AXI GP0 interface**.

#### Add AXI Interconnect

1. Add an **AXI Interconnect** block.
2. Set **Number of Master Interfaces** to 1 (connecting to the Beamformer IP).

#### Add the Beamformer Subsystem IP

1. Click **+** → search for `Subsystem_ip` (or the name of your generated IP).
2. Add it to the canvas.
3. The IP should expose AXI4-Lite slave ports (`S_AXI`, `S_AXI_ACLK`, `S_AXI_ARESETN`).

#### Add Processor System Reset

1. Add a **Processor System Reset** block.
2. This takes the raw `FCLK_RESET0_N` from the Zynq PS and produces synchronized reset signals for all PL blocks.

### Connect the Blocks

Click **Run Connection Automation** (select all) — Vivado will auto-connect most signals. Verify the following connections manually:

| Source | Destination |
|---|---|
| `ZYNQ PS.FCLK_CLK0` | `AXI Interconnect.ACLK`, `Subsystem_ip.IPCORE_CLK`, `PS Reset.slowest_sync_clk` |
| `ZYNQ PS.FCLK_RESET0_N` | `PS Reset.ext_reset_in` |
| `PS Reset.peripheral_aresetn` | `AXI Interconnect.ARESETN`, `Subsystem_ip.IPCORE_RESETN` |
| `ZYNQ PS.M_AXI_GP0` | `AXI Interconnect.S00_AXI` |
| `AXI Interconnect.M00_AXI` | `Subsystem_ip.AXI4_Lite_Slave` |

### Validate the Design

1. Click **Validate Design** (Ctrl+Shift+V).
2. Resolve any critical warnings or errors.
3. Save the block design.

### Create HDL Wrapper

1. In **Sources** panel, right-click on the block design file (`.bd`) → **Create HDL Wrapper**.
2. Select **Let Vivado manage wrapper and auto-update**.
3. A top-level wrapper file (`beamform_wrapper.v`) will be created.

#### 1. Imports and Overlay Loading

```python
from pynq import Overlay, MMIO
import matplotlib.pyplot as plt
import numpy as np
import math

# Load the bitstream onto the FPGA Programmable Logic
print("Loading overlay...")
overlay = Overlay("beamform_wrapper.bit")
print("Running continuous sine wave through the FPGA fabric...")
ip = MMIO(0x43C00000, 0x10000)   # AXI4-Lite base address of the Beamformer IP
```

The `Overlay()` call dumps the `.bit` file into the PL fabric and parses the `.hwh` file to understand the address map.

#### 2. Helper Functions

```python
def float_to_sfix16_en12(val):
    """Convert Python float to sfix16_en12 fixed-point integer."""
    max_val = (2**15 - 1) / 4096.0
    min_val = -(2**15) / 4096.0
    val = max(min_val, min(max_val, val))
    int_val = int(round(val * 4096))
    if int_val < 0:
        int_val = int_val + (1 << 16)
    return int_val & 0xFFFF

def sfix16_en12_to_float(val):
    """Convert sfix16_en12 fixed-point register value back to float."""
    if val >= 0x8000:
        val -= 0x10000
    return val / 4096.0

def write_complex_input(base_addr, real_val, imag_val, strobe_addr):
    """Write a complex fixed-point value to the IP via AXI4-Lite."""
    ip.write(base_addr + 0x04, float_to_sfix16_en12(real_val))
    ip.write(base_addr + 0x08, float_to_sfix16_en12(imag_val))
    ip.write(strobe_addr, 1)

def read_complex_output():
    """Read the combined beamformed output from the IP."""
    return (sfix16_en12_to_float(ip.read(0x148)),
            sfix16_en12_to_float(ip.read(0x14C)))
```

#### 3. Configure the Steering Angle

```python
# Enable the IP
ip.write(0x04, 1)   # Enable IP

# Steer the beam to 30 degrees
steer_angle_deg = 30
phase_shift = math.pi * math.sin(math.radians(steer_angle_deg))

# Write the steering angle to the IP register
ip.write(0x10C, float_to_sfix16_en12(math.radians(steer_angle_deg)))
```

#### 4. Time-Domain Simulation and Hardware Execution

```python
time_steps = np.linspace(0, 4 * math.pi, 150)

in1_plot, in2_plot, in3_plot, in4_plot, out_plot = [], [], [], [], []

for t in time_steps:
    # Generate 4 phase-shifted sine waves (amplitude = 0.5)
    sig1_r = 0.5 * math.cos(t - 0 * phase_shift)
    sig1_i = 0.5 * math.sin(t - 0 * phase_shift)
    sig2_r = 0.5 * math.cos(t - 1 * phase_shift)
    sig2_i = 0.5 * math.sin(t - 1 * phase_shift)
    sig3_r = 0.5 * math.cos(t - 2 * phase_shift)
    sig3_i = 0.5 * math.sin(t - 2 * phase_shift)
    sig4_r = 0.5 * math.cos(t - 3 * phase_shift)
    sig4_i = 0.5 * math.sin(t - 3 * phase_shift)

    # Write inputs to FPGA via MMIO
    write_complex_input(0x100, sig1_r, sig1_i, 0x118)
    write_complex_input(0x10C, sig2_r, sig2_i, 0x120)
    write_complex_input(0x118, sig3_r, sig3_i, 0x128)
    write_complex_input(0x124, sig4_r, sig4_i, 0x130)

    # Read the real-time hardware output
    r_real, r_imag = read_complex_output()

    # Store results for plotting
    in1_plot.append(sig1_r)
    in2_plot.append(sig2_r)
    in3_plot.append(sig3_r)
    in4_plot.append(sig4_r)
    out_plot.append(r_real)
```

#### 5. Plot Results

```python
plt.figure(figsize=(12, 5))

# Plot individual input waves (dashed)
plt.plot(time_steps, in1_plot, linestyle='--', alpha=0.7, label='Antenna 1 Input')
plt.plot(time_steps, in2_plot, linestyle='--', alpha=0.7, label='Antenna 2 Input')
plt.plot(time_steps, in3_plot, linestyle='--', alpha=0.7, label='Antenna 3 Input')
plt.plot(time_steps, in4_plot, linestyle='--', alpha=0.7, label='Antenna 4 Input')

# Plot the combined FPGA output (bold)

plt.title('Time-Domain Sine Waves (Hardware Output)', fontsize=14, fontweight='bold')
plt.xlabel('Time')
plt.ylabel('Signal Amplitude', fontsize=12)

```

### AXI4-Lite Register Map

| Offset | Register | Description |
|---|---|---|
| `0x04` | IP Enable | Write `1` to enable the beamformer core |
| `0x100` | Antenna 1 Real | Fixed-point real part of antenna 1 input |
| `0x104` | Antenna 1 Imag | Fixed-point imaginary part of antenna 1 input |
| `0x10C` | Antenna 2 Real | Fixed-point real part of antenna 2 input |
| `0x148` | Output Real | Fixed-point real part of beamformed output |
| `0x14C` | Output Imag | Fixed-point imaginary part of beamformed output |

> **Note:** Exact register offsets depend on how HDL Coder assigned the AXI4-Lite interface. Check the generated `component.xml` or the PYNQ overlay's `.hwh` file for the definitive register map.

## 14. Troubleshooting

### HDL Coder: "Block is not supported for HDL code generation"

**Cause:** A block inside the subsystem does not have an HDL implementation.
**Fix:** Replace with an HDL-compatible equivalent. Common substitutions:
- `MATLAB Function` block → `Gain`, `Product`, `Add`, `Delay` blocks
- Floating-point blocks → Use **Data Type Conversion** to `sfix16_en12` first
- `Scope` → Remove from subsystem (keep in testbench only)

### HDL Coder: Timing Violations After Synthesis

**Cause:** Logic path is too long for the target clock frequency.
**Fix:**
- Add pipeline registers in Simulink using the **Delay** block
- Enable **pipeline registers** in HDL Coder options under **Optimization** → **Pipeline Registers**
- Reduce target clock frequency in the Workflow Advisor

### Vivado: IP Not Found in Catalog

**Cause:** The `ip_core/` path was not added to the Vivado IP repository.
**Fix:** In Vivado → **Tools** → **Settings** → **IP** → **Repository** → add the full path to `ip_core/`.

### PYNQ: `Overlay()` Raises `RuntimeError`

**Cause:** The `.bit` and `.hwh` files are in different directories, or the filenames do not match.
**Fix:** Ensure both files share the same base name and are in the same folder as the notebook.

### PYNQ: Output Always Zero

**Cause:** The IP Enable register was not written, or the AXI base address is wrong.
**Fix:**
- Write `ip.write(0x04, 1)` to enable the IP before writing inputs.
- Verify the base address by checking `overlay.ip_dict` in Python — it lists all IP cores and their addresses.

### PYNQ: Output Looks Noisy

**Cause:** Reading output before the IP has processed the inputs (latency).
**Fix:** Add a small delay between writing inputs and reading output:
```python
import time
write_complex_input(...)
time.sleep(0.001)  # 1 ms delay
r_real, r_imag = read_complex_output()
```
